# API 框架升级改造清单（通用施工版）

适用范围：

- .NET / Node.js / Java 等后端 API 项目
- 使用数据库、缓存、对象存储、短信、支付或第三方 API 的业务系统
- 需要从“能跑”升级到“可维护、可部署、可公开审计”的项目

本文只提供通用治理清单，不包含私有业务数据、真实接口地址、真实密钥或生产配置。

## 1. 安全治理

### 1.1 Token 有效期

目标：

- 签发和校验使用同一时间标准。
- 不因日期截断、时区或反向判断导致会话异常失效。

要求：

- 推荐使用 UTC。
- 校验逻辑表达为“当前时间早于过期时间”。
- 前端自动清登录态逻辑与后端 401/403 保持一致。

验收：

- 登录后 token 在预期有效期内可用。
- 过期 token 被稳定拒绝。

### 1.2 敏感参数传输

目标：

- 密码、验证码、token、密钥不出现在 URL、浏览器历史、反向代理日志或普通请求日志中。

要求：

- 敏感操作使用 `POST + Body`。
- DTO 做必填、长度、复杂度或格式校验。
- 日志中对敏感字段脱敏或跳过 body 记录。

验收：

- URL 中无密码、验证码和 token。
- 日志中无明文敏感字段。

### 1.3 测试能力隔离

目标：

- 生产环境不存在认证绕过和测试通道。

要求：

- 删除或隔离模拟登录、万能验证码、测试手机号白名单、未鉴权测试发送接口。
- 保留测试能力时必须限制到 Development 环境，并使用显式开关。

验收：

- 生产配置下无法匿名获得登录态。
- 生产配置下无法调用未授权测试发送接口。

## 2. 配置治理

### 2.1 Secrets 分层

目标：

- 仓库可公开，真实密钥不进入 git。

要求：

- 公开配置只保留结构、默认值和占位符。
- 私有配置进入 secrets 文件、环境变量或部署平台密钥管理。
- `.gitignore` 覆盖所有本地 secrets 文件。

验收：

- 敏感词扫描只命中文档说明、占位符或字段名。
- 部署环境可读取私有 secrets。

### 2.2 连接串治理

目标：

- 数据库、Redis、Mongo、对象存储等连接信息不硬编码在代码中。

要求：

- 连接串从配置注入。
- 本地开发、测试、生产配置分离。
- 缺少必要配置时启动阶段明确失败。

验收：

- 移除代码内默认真实连接串。
- 配置缺失错误可读。

### 2.3 CORS

目标：

- 避免任意来源携带凭证访问 API。

要求：

- 使用来源白名单。
- 管理端、公开端、回调端按需拆分策略。

验收：

- 非白名单来源被拒绝。
- 合法来源正常访问。

## 3. 稳定性治理

### 3.1 缓存降级

目标：

- 缓存故障不拖垮核心业务。

要求：

- 读缓存失败回退数据库或默认值。
- 写缓存失败只记录 warning。
- 缓存 key 有统一命名规则。

验收：

- Redis 不可用时核心读写路径仍可运行或明确降级。

### 3.2 系统配置初始化

目标：

- 空库或新环境首次保存配置不失败。

要求：

- 不存在配置记录时自动创建。
- 默认值集中定义。

验收：

- 空配置表环境下可首次保存并读取配置。

### 3.3 外部服务容错

目标：

- 天气、短信、地图、支付、对象存储等外部服务失败时，业务有明确降级。

要求：

- 设置超时。
- 区分可重试和不可重试错误。
- 记录必要上下文但不输出密钥。

验收：

- 外部服务失败不会导致无关页面或接口 500。

## 4. 可选业务能力

### 内容系统

- 支持多栏目关联。
- 公开查询按栏目筛选。
- 详情返回关联栏目。
- 首页展示配置从后端读取。

### 短信/通知系统

- 提供统一发送入口。
- 支持供应商切换。
- 群发任务要去重、校验、限流。
- 发送记录区分提交态、成功态、失败态。

### 公开展示系统

- 公开接口返回空态而不是假数据。
- 前端显示字段与后端 DTO 对齐。
- 外部数据使用缓存与降级。

## 5. 交付检查

每个模块完成后至少检查：

- 构建通过。
- 关键接口 smoke test 通过。
- 无真实密钥进入 git。
- 日志无明文敏感字段。
- 配置缺失时错误明确。
- 新增配置项有文档。
- 有兼容风险时写入交付说明。
